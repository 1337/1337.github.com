When to Use Unicode Strings

Still stuck on Python 2.x? Bad news, strs are not unicodes. That means you
will need to know when to use unicode and when to handle str. Should you
use unicode? When should you use it? Are there gotchas? Let’s find out.

What is a Python unicode?

Let’s first establish what a str is.

strs are… strings of bytes. They have series of 1 and 0, encoded in
a format defined when the string was generated.

unicode, on the other hand, abstracts the concept of bytes away from the
programmer, so byte-counting is no longer necessary.

Encoding

Since strs are just series of bytes, there is no limit to what they can hold.
A string can contain “unicode characters” as we understand it, because “unicode
characters” are, also, series of bytes.

What is str.encode()?

str.encode() does nothing.

What is unicode.encode()?

unicode.encode() always returns a str.

Decoding

What is str.decode()?

str.decode() always returns a unicode.

What is unicode.decode()?

unicode.decode() does nothing (if the encoding is utf-8).

Why should strings be encoded and decoded?

It is important to decode your str to a more specialised instance as soon
as you got it, because unicode counts characters properly. For example, in
this example, "Krstić" appears to have six

Implicit conversion

Mixing str and unicode automatically decodes the str to unicode
using the default system encoding, which is usually 'ascii'.

This means decoding errors can occur if the string is not encoded in UTF-8.

When to use unicode

tl;dr Almost Everywhere. With the exception of files, just remember:

  • decode (to unicode) early
  • use unicode where the text is meant to be displayed
  • encode (to str) only when necessary

ASCII strings

YES! With the exception of internal strings (e.g. keys), there are no
drawbacks to using unicode strings for storing ASCII characters.

A caveat to doing this is if there is any poorly-written code that explicitly
checks if isinstance(your_string, str), in which case, using unicode is
not an option, unfortunately.

It is not advisable to compare str instances with unicode instances
with the == operator. The example there is just for reference.

Dictionary keys

Similarly, any ASCII dictionary keys are directly accessible with its unicode
accessor, and vice versa, but unless your dict keys will be seen by the user
(which is possible in some cases), you should not be doing it.

Implementation

Although there is a bytes thing in python2, it was not implemented
properly. Python3 implemented it properly, but not for python2.
What this means is although bytestrings should be used for dict keys in
python3, this does not concern python2.

somepackage.some_function(str)

If a function says it takes a str, pass a str. Otherwise, hope for the best
and assume duck typing works on the unicode.

If you pass unicode into a function you do not control, be ready to
“be in for a world of hurt“.

Logging

There is no difference, it seems, between logging strs and unicodes,
even when the unicode string is %s‘d into a standard string.

Printing

The fact that a str can contain unicode characters confuses me, but the
rule of thumb is: if someone is going to read this text, it should be unicode.

Anything calling unicode(variable)

Watch out when using a function, or a package function, that explicitly calls
the unicode() function, because passing unicode into unicode() fails
ironically.

This is because unicode() accepts a str as its first argument. To confuse

Reading and writing to a file

When files are concerned, using str (remember, series of bytes) is much more
intuitive than using unicode, which can lead to undesirable errors.

__main__

You can use unicode for __main__, but you shouldn’t; this is largely due
to the fact that there are no unicode characters in "__main__" to begin with,
and that __main__ is not user-facing text.

Python’s file() selected 'ascii' because python2’s default encoding is
'ascii', making it impossible to write some unicode instances directly.

from __future__ import unicode_literals

unicode_literals changes all instances of 'strings' in a module to
type unicode. This seems harmless at first, but it also changes the return type
of all string functions in a module, making it hard to debug.

Avoid unicode_literals. This is also the explicit unicode literals PEP
passed, reintroducing u'unicode literals' back in Python 3.3+.

To handle unicode in projects that support both python2 and python3, investigate
the use of six:

When using Django

Django 1.5+ provides various helper functions for handling unicode in your
project. They include @python_2_unicode_compatible,
which automatically adds __unicode__ to any model with __str__, and
smart_text (lazy) / force_text (non-lazy), which guarantee to return
unicode instances.

six is installed as a dependency.

When a str is not necessarily encoded in UTF-8

Just because UTF-8 gets all the love, it does not mean all strings are encoded
in it. For example, PHP defaults to UTF-16, which would be wrong to decode
to a UTF-8 unicode instance. To guess the encoding of something, use
chardet.detect().

Again, decode as early as possible. Passing a string of unknown encoding out
of the data model will inevitably cause problems when it needs to be spliced.

When UTF-32 is used

Python2 does not support UTF-32.

The Official StartupCompany Guide

Welcome to StartupCompany. In this guide, you will learn what is necessary to work with StartupCompany as a developer.

  1. You will be assigned tasks to work on. As our development is results-driven, you will not be given deadlines until the due date.
  2. The business team makes decisions based on the team’s intuition. You will be notified of their intuition when the client demands something.
  3. Often, deadlines may be from an earlier point in time.
  4. We run our in-house Test-Driven Development (TDD). All our tests are currently passing – new team members are expected to write zero tests.
  5. StartupCompany. frequently gives away free trials to customers. The conversion rate of these trials has so far been 0%. Your responsibility is to make these free trials a reality.
  6. As a form of money management, the CEO will pay you with money that will eventually exist in six-month’s time. If there is no money in the bank before the sixth month, you will not be paid.
  7. As an expert in the field, you will be expected to write code that works from IE7 to IE12, including development and debug builds of desktop and mobile browsers.
  8. Every developer is encouraged to document their code and processes. You can find our large repository of outdated docs in our FreeProduct repository.
  9. Here at StartupCompany, we strive for results. Using the Thick Loaves(TM) philosophy, you are encouraged to push code to production every 5 minutes, however stable it is, as long as it is stable. (see point 7)
  10. Employees are expected to complete tasks in under 50% of the allocated time.
  11. We have many clients, and, as such, have to fulfill requirements from each client. You will be creating marvelous work of art that does all that with just one code base.
  12. You, as a new member at StartupCompany, are very special. We treat you like a team. The whole team.
  13. We use Dipchat as our company issue tracker. If you find a bug, please file it inside the room named ‘Developers’. The bugs that you file will not be resolved; the bugs that you address will not be acknowledged.
  14. The Business team has no technical knowledge related to building and designing a product. As such, if they ask you how long something would take to implement, please work with them and reply with ‘a very short time.’
  15. We are a six year-old company, and we have made a lot of great products in the past. You will be working on a relatively new project that needs to have features from all previous iterations.
  16. After reading about Exception-driven development (EDD) on Hacker News, we have adopted it as a practice. The moral of the story here is that if we monitor our systems, then they will not fail.

Medicals

Let me tell you a story about Medicals, my personal project from 2004.

Prequel

I did an educational manoeuvre at around the age of 14. I was an innocent person at the time – so young, so ambitious, yet so full of nothing. My life took a strange turn one day, when the school where I studied decided to give us all a project in Microsoft Access:

Make a form that shows contents of a table. (or something… it was more than a decade ago)

Intuitively, all that a student had to do was to fire up MS Access, create a new database with a form in it, where fields in the form are linked to columns in that table. That would get you 100%. However, being the person that I was, I made such a thing:

(The ‘table’ beneath the UI handled expense records, hence the name “Expense”.)

From the screenshot, it was apparent that I made at least 5 iterations of the Expense project.

David Kirby, my ICT (IT) teacher at the time, told me I did a good job.

I did a good job.

At the age of (a young age), my naivete thought that MS Access was the go-to platform for creating computer interfaces, for realising dreams like others do; ranging from classic address books to entire operating systems, I attempted recreating them all.

FA, Windows 98 in Access!

FA, Windows in Access run on Windows that looks like Mac OS X!

FA, Windows XP in Access – it’s even widescreen!

These were all created at the peak of my ignorance. At that age, when the Internet was nothing more than Yahoo! mail, MSN Explorer, and stickdeath.com, ignorance is not only bliss–it was my only choice. To make sure you understand how stupid these projects were, everything in the form was an image resource. Each resource points to an image file, which then in turn can be placed on the form. They have pre-written handlers that simply make other elements show and hide.

Then came Medicals.

Medicals: the early days

At the height of passionate “software” development, I came up with this ambitious idea while visiting my family doctor, who went by the name of Ng. He was an incompetent wad who happened to be the doctor closest to where my family resided. He had cabinets upon cabinets of patient records that were retrieved by hand. I thought: what if I made something for this guy, who clearly has a need for a computerised system?

Having gained the experience to do stupid things very quickly, Medicals 1.0 (not pictured) was born. The first version, like its predecessors, are merely MS Access forms that link to table entries. However, Medicals 1.3 was a little different. Forms were linked to queries instead, and there was a reporting option, which were reports linked to queries. This allowed the interface to show both information about a particular patient, and a list of medications that was given to the patient (in the image above, the patient was repeatedly given arbitrary doses of Vitamin C).

Medicals 1.5 was “released” in the sense that medicals.mdb was copied elsewhere as medicals15.mdb. From the only surviving screenshot, it can be seen that doctor Ng can now record the number of days for which he prescribed water to the patient.

Medicals 2: where the crazy got crazier

Even a crazy person would, at this point, realise that real people, like a doctor, would not use something, let alone a mission-critical database, from a (young person). However, I was not crazy. I was deranged. I was an energetic, deranged man, who would take this project to the end of its days, until someone, somewhere, actually uses my work for real.

The Medicals 2 series was made “available to the public” some time in 2005.

Version 2 underwent a significant visual overhaul (read: turned buttons into images), a change of identity (that bogus company from previous screenshots had been swapped with yet another bogus one), as well as a few for loops, evident in the “Automate/Batch” section. I have no clue what it did anymore.

One key innovation I thought I implemented at the time was the concept of loading. Many software programs at the time – particularly Microsoft products – had loading screens, which was shown while the software did something behind the scenes.

I also made a loading screen for Medicals. Not before the database was slow, but because I wanted a loading screen. As a result, the loading screen was simply opened and closed according to a timer.

Medicals 2.65: where the talking stops

Medicals 2.49 was the last version with a preserved screenshot, and the final version number for the project went as high as 2.65. I discovered something called Visual Basic, which, behold, created real software executables. That, along with whatever reason that prevented databases created by non-English versions of MS Access to be opened elsewhere, ended my plan to ever re-open the archive of mystery that is Medical265.mdb.

Learning How Not to Fail

Startups (start-ups) are hard.

You are the CEO of QuickHolland, a company that sells instant hollandaise sauce (currently in R&D) in revolutionary, squeeze-bottle form factor.

As a founder in a startup, you likely have several priorities to juggle:

  1. Maintaining your vision, because it is your company
  2. Maintaining investor relations, to get advice from them (if they are on the board) or to have a constant source of funding
  3. Managing your day-to-day business, because, again, this company is yours to run.

This post explains why it is often difficult to do all of these things.

1. Maintaining your vision

Your mother just spent an hour making hollandaise sauce from scratch, and she complains to you that it was too time-consuming a thing to do in the morning–when people need to get to work on time.

You have an idea; a lightbulb glowed atop your head. Instant hollandaise sauce. You narrow your vision. You set your goal. You incorporate your company. And you hire on the best people you know.

Now what?

Is it your job to stick to the vision, or is it sometimes appropriate to modify the company’s vision?

One year into your saucy enterprise, you find yourself in a sticky situation, where your food scientists come up with something in a squeeze bottle, but no one is buying it. “But build it and they will come,” you think to yourself, “maybe I am not reaching the correct market.”

You pitch your idea to your investors and they say that your idea reeks of unoriginality. They say that it has been done before. “Is this like other hollandaise sauces out there, except on social networksin a squeeze bottle?” You tell them yes, but, see, the squeeze bottle is what we’re really selling. Your investors, agreeing with your explanation, throw money at you for another year of runway.

Are you now selling hollandaise sauce, or are you now selling squeeze bottles?

You go back to your team of scientists and tell them you have decided to pivot. To pivot is to change the direction of the company–your corporate vision has now changed. Half of your team leaves your company because they are food scientists, not petroleum engineers (who would be the people who commonly make squeeze bottles). “No matter,” you think to yourself, “I have leads.”

2. Maintaining investor relations

You own 100% of the company, but you need some initial funding, so the first course of action, of course, is to ask family and friends for your “bootstrap” money. That money buys you time, the ability to pay lawyers and governments for business-related fees, and money to hire your first employee.

Of course, you will almost certainly want to hire more than one employee, so you sell your company’s shares, to your investors (wherever you got hold of them to begin with), for instant funding, paid back by them as convertible notes. With that capital, you hire your second, third, fourth employees, and so on. You know how long your runway is (how much money you have {div} how much money you spend per month), and so do your investors, because they request that kind of information. They give you funding because they know you have a team who can execute your idea. They give you funding because you seem like a competent leader. They give you funding because, ultimately, your company might succeed, which means more money for them.

What if you don’t succeed?

3. Managing your day-to-day business

QuickHolland is your first business, and it is your baby. By now, you have been working on it for three years, and you have pivoted three times so far; your company now prints posters for supermarkets that sell your brilliant squeeze bottles, turning your company into an advertising agency. You still have your brilliant team of people, but only one food scientist is left at this point, because the company isn’t really making food anymore.

On your fourth year, you run out of funding (which is a surprising amount of time, because investors gave you only one year’s worth of money.) You cannot pay your employees for the time being, but, no matter, this too shall pass. You try very hard to look for new investors. You get about half a year worth of runway, yet again, saving the company once more.

Are you going to pay your employees their unsettled wages? If so, how?

Once, while swimming in Salt Lake, you encounter a young, attractive woman, who happens to be interested in you. You two share the same interests, same backgrounds, and quickly fall in love. Once she moves in, You quickly realise that you no longer have the time and patience to run and maintain the company that was once your baby and vision. Hell, it isn’t even what your mother asked for anymore.

What comes first? Your life, or your company?
