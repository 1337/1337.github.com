Python Core Devs Need You

Part of the PyCon series, paraphrasing whatever Raymond Hettinger said that day.

The core developers are not normal human beings

And they need to hear from you.

They just make the language and the standard library; they don’t know your use cases.

One time, a kind of averaging function made it into the standard library. It calculated the average to a billion digits of precision.

Well, as it turns out, that sort of precision isn’t meaningful to anyone using that function, so time is simply wasted calculating it.
If it is slow, then people won’t use it.
If no one will use it, then why is it in the standard library?

Python should be more friendly, and he could not count the ways

Throw a timestamp into a datetime function, and it says Expected a frggn buffer object. What does that even mean? You (the ordinary person) can definitely write more meaningful exception messages for the standard library.

How much better would it be, and how much time would it save everyone, if every error had a format like

That helps everyone, especially beginners.

The language is 25+ years old

list() works on a file but not on StringIO. Why is that? It is not intentional; the developers just didn’t have the time to add an equivalent to StringIO. If you want to help, this is one thing you can do.

Help update the docs

There are still parts of the standard library that are lacking docstrings. If you can accurately document these libraries, by all means!

Every time you submit even a trivial “fix”, it takes core devs an enormous amount of time

Just because you (the new developer) think something looks off, it doesn’t mean it should be “fixed”.
One example of such a story he told was someone changing all unit tests doing assert ... > ... to self.assertGreater(..., ...). Unbeknownst to the dude, some of these tests are actually supposed to test the bitwise shift operator (>), making some of the tests useless.
He also cautioned against refactoring other people’s tests. While code has tests to guard against mistakes, tests themselves do not, especially when some code was red-green developed, and refactoring the tests can potentially reduce the test coverage, or even give out false results.

It is pronounced the tupple

At least he’s saying it that way.

Advice for new developers

If you want to help, go do code reviews on the (Python) bug tracker or PEPs.

PEPs and bug trackers need critical thinking (i.e. knowing why something should not be implemented, rather than why something should).

Every Python feature request should result in a net reduction in complexity. It is easy to implement something. It is hard—extremely hard—to justify not implement it.

Work with developers who talk more about what they don’t know than what they do know.

If you learn without getting the impact, you’re working out wrong.

Understand what problems people are trying to solve, not what frameworks people are throwing at them all day long.

Pragmatic Microservices for the Rest of Us

Part of the PyCon series, by Ryan Easterbrook.

“Well-made microservices can be rebuilt in a day”

The difference between services and microservices is that microservices only take care of one thing. If a service takes care of more than one thing, then it becomes harder to manage.

Microservices also tend to share the same standardised interface, which also help with managing them.

Example interface: REST

A simple microservice

A simple microservice consists of:

  • A shared “service base”, where all the boilerplate code is stored,
  • Service implementation, and
  • Protocol handler.

Database

None of the microservices will share the same database table.

If microservice A wants something from table B, it will ask microservice B for the information.

None of the microservices will share the same database table. This makes migrations massively easier to create, rollback, and squash. Having none of the microservices share the same table also means these tables can be taken out of the database with no difficulty, and can be scaled up using another instance very quickly.

Foreign keys

There will be no foreign keys. The code must manage the integrity.

Error handling

All microservice interfaces must be hard and well-defined.
If microservice A calls microservice B calls microservice C, and C raises an error (e.g. HTTP 500), then B must first catch all exceptions, and decide for itself whether to retry calling microservice C, or fail itself to microservice A as well.

Under no circumstance should microservice A receive any uncaught exceptions from microservice C.

Logging

Request logging is going to be easier per-microservice, but these logs themselves will make no sense. So, service request logs should be a thing.

Service Request Logs are trees of logs gathered to serve a single (web) request.

Having these logs also helps with performance tracking, to troubleshoot which microservices fail more often than expected.

Justification for microservices

Use microservice architecture if

  • you have multiple clients (or demos)
  • you have many developers, so they don’t need to rebase over each other’s stuff, or fight to merge into HEAD
  • you need to keep codebase complexity down, once and for all..

Arguments against microservices

  • Microservice code is cleaner only if your developers have the ability to write proper code. “Normal services” already encourage cleaner code, if developers have discipline.
  • Distributed services are never easy. Distributed transactions are even harder. No one will recover these errors for you, so you will need to manage each and every case.
  • Microservices are always slower than pure function calls, or a lower-numbered, normal services. If you want speed, try another language.
  • Testing microservices is easy. Testing integration among microservices just became massively more difficult.

Version Control Worst Practices

Part of the PyCon series, by Greg Ward.

Commit any old crap

  • Sources of your libraries…
  • Built artifacts…
  • Build outputs…
  • Database from production…

Using version control as a backup tool

  • Except you backup (“commit”) whenever you want, not on a regular interval
  • Except you can ignore files for some reason
  • Except you need a message to justify why you’re backing up…

File sharing

  • Need to share a file? Commit it and push it on Github, so someone can pull it down!

Lone Wolf

  • Some developers can be such powerful version control users that their setup can discourage their setups from being used by any others.

Terrible commit messages

  • WIP
  • ongoing dev
  • .
  • working, pretty close (please come back)

Commit broken code

  • “Because it’s a backup tool! Better commit now!”

Commit nothing

  • “I can’t commit broken code!”

Committing unrelated changes

  • Switching branches / splitting changes is just too hard
  • Related: squashing changes

This makes commit history hard to understand.

Monolothic repository

  • Forcing your entire team (or worse, company) to share the same release schedule, version numbering, branching policy…
  • Even more fun when two teams want to merge into HEAD at the same time
  • Rebase hell

Too many repositories

  • One repository for each microservice is just too many
  • Should always put boilerplate code in its own repository, not have a copy in each repo

Presume infinite scalability

  • Remember version control systems also take time to do things
  • More commits means more time
  • Bigger files means more space

Securing Your Company's Data

Part of the PyCon series, by Elissa Shevinsky.

The Five Principles of Data Security

  1. Minimise Collection: Don’t keep anything you don’t plan to use.
  2. Encrypt what you must keep
  3. Delete what you don’t need: And don’t just mark things as deleted, either; that won’t fool hackers / the government.
  4. Have gates on data access: Especially internally.
  5. Empower your CSO: Let the CSO veto marketing decisions.

Threat modelling

Threat modelling concerns the source of the risk. Different kinds of risks require different kinds of protection.

Who is your target:

  • Your mother?
  • The government?
  • A black hat?
  • A coworker?
  • Muslims?

Delete what you don’t need

Data is a liability, and companies should know that it is a liability to store customer data.
When you lose customer data, you will get sued.
So while data is valuable, it might not be worth the liability to store it.

Before you store customer data: what can you not collect from your users?

Some examples include:

  • Asking for the postal code when someone gets their groceries
  • Asking for phone numbers when someone gets a haircut
  • Turning on geolocation permission just to auto-fill a city-level location box

Have gates on data access

Most data breaches occur from within the company. Give employees only enough access/permissions to do their job.

Empower your CSO

The Chief Security Officer is supposed to manage the security aspect of your company, product, and projects.
If you don’t give it any power, why hire one?

When marketing says it wants to install analytics on your software, the CSO needs to make a call on whether it is justified or overkill for the purpose.
