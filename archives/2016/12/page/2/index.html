Version Control Worst Practices

Part of the PyCon series, by Greg Ward.

Commit any old crap

  • Sources of your libraries…
  • Built artifacts…
  • Build outputs…
  • Database from production…

Using version control as a backup tool

  • Except you backup (“commit”) whenever you want, not on a regular interval
  • Except you can ignore files for some reason
  • Except you need a message to justify why you’re backing up…

File sharing

  • Need to share a file? Commit it and push it on Github, so someone can pull it down!

Lone Wolf

  • Some developers can be such powerful version control users that their setup can discourage their setups from being used by any others.

Terrible commit messages

  • WIP
  • ongoing dev
  • .
  • working, pretty close (please come back)

Commit broken code

  • “Because it’s a backup tool! Better commit now!”

Commit nothing

  • “I can’t commit broken code!”

Committing unrelated changes

  • Switching branches / splitting changes is just too hard
  • Related: squashing changes

This makes commit history hard to understand.

Monolothic repository

  • Forcing your entire team (or worse, company) to share the same release schedule, version numbering, branching policy…
  • Even more fun when two teams want to merge into HEAD at the same time
  • Rebase hell

Too many repositories

  • One repository for each microservice is just too many
  • Should always put boilerplate code in its own repository, not have a copy in each repo

Presume infinite scalability

  • Remember version control systems also take time to do things
  • More commits means more time
  • Bigger files means more space

Securing Your Company's Data

Part of the PyCon series, by Elissa Shevinsky.

The Five Principles of Data Security

  1. Minimise Collection: Don’t keep anything you don’t plan to use.
  2. Encrypt what you must keep
  3. Delete what you don’t need: And don’t just mark things as deleted, either; that won’t fool hackers / the government.
  4. Have gates on data access: Especially internally.
  5. Empower your CSO: Let the CSO veto marketing decisions.

Threat modelling

Threat modelling concerns the source of the risk. Different kinds of risks require different kinds of protection.

Who is your target:

  • Your mother?
  • The government?
  • A black hat?
  • A coworker?
  • Muslims?

Delete what you don’t need

Data is a liability, and companies should know that it is a liability to store customer data.
When you lose customer data, you will get sued.
So while data is valuable, it might not be worth the liability to store it.

Before you store customer data: what can you not collect from your users?

Some examples include:

  • Asking for the postal code when someone gets their groceries
  • Asking for phone numbers when someone gets a haircut
  • Turning on geolocation permission just to auto-fill a city-level location box

Have gates on data access

Most data breaches occur from within the company. Give employees only enough access/permissions to do their job.

Empower your CSO

The Chief Security Officer is supposed to manage the security aspect of your company, product, and projects.
If you don’t give it any power, why hire one?

When marketing says it wants to install analytics on your software, the CSO needs to make a call on whether it is justified or overkill for the purpose.
