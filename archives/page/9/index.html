Code Review Checklist

You can’t keep 100+ items in your head as you look at every method or clause in a code base, so you’re going to have to read the code over and over, looking for different things. - Smartbear.com

Apart from static checking, which your automation tools should already do, there are always a set of things to watch out for when you review your own, or others’ code.

Semantics

Does it break any rules?

The bronze rule of programming is minimising code complexity. Code is more often run than read, which is why it must be optimised for speed Code is more often read than written, which is why it must be optimised for reading. And in order for code to be easy to read, it does not matter how short or long or idiomatic it is, if it does not conform to what the rest of the codebase is doing.

For example: if the rest of the code uses CBV and you write a FBV because “flat is better than nested”, you are forcing the rest of the team to make a mental exception. And mental exceptions are very, very expensive.

Another example: if your database stores all masses in kilograms, and you swoop in and add two fields, one in grams and one in tonnes… although both units are SI (and therefore kosher), other coders must make mental exceptions for these fields. “Using the most appropriate unit” is not special enough to break the rules.

You may also be one of the aspiring interns in your company who know a lot about naming conventions in PHP (documented in whichever PSR), so when it is your turn to make an impact, you write your classes in StudlyCaps and all their methods camelCase, even though all other code is already written like_a_perl_god_did_it. You are technically correct; it is not the right time and place.

Is this good code?

Just because you maxed out your static checks and lowered your max cyclomatic complexity to 0 (nb. this doesn’t work) and the code passes all tests and static checks, it doesn’t mean the code is good. In fact, if you improperly configured your static checks, your developers can occasionally write code that is even uglier that it needs to be, just to bypass the draconian checks that you have imposed. For example, if every function is limited to 5 lines max, then there will be a lot of incomprehensible one-liners.

Can I easily explain this code to someone else?

Explaining your own code to someone else is easy: you wrote it; you know the edge cases. What is much harder (and makes code reviews important) is whether you can review someone else’s code, and understand it well enough to relay your knowledge. It requires both you to have read the review thoroughly, and the code to be easy to understand.

If the implementation is hard to explain, it’s a bad idea.

Should this code be here?

Bob wrote a Haversine function for calculating the distance between two lat/long pairs. Neat. He submits it for review. It passes all tests and builds. What would you do?
If your code is only meant to deal with lat/long pairs less than 100 metres apart, you are wasting your time; you are wasting the machine’s time. The code can be replaced with a middle school formula.

Languages and Tools

JavaScript

  • Are the docstrings actually correct?
  • Is any code using the side effect of an assignment as a value?
  • Are promises properly written, not in the manner of a callback hell?
  • Are finally promises written as ['finally'], so that IE8 doesn’t register it as a keyword error?
  • Any unnecessary arguments being passed to any function/method? (Not the static check kind of unnecessary—the law of demeter kind of unnecessary.)

Angular 1

  • Is there strict-di?
  • Are controllers referenced using controllerAs?
  • Are services written using the this keyword?
  • Are custom-made scopes being destroyed manually? Can they be garbage-collected?

Python

  • Are any of the default function/method arguments mutable?
  • Is print() being used in production code?
  • If you wrote a function named is_... that returns anything other than True or False, did you ask yourself what you were doing?
  • Is everything public type-hinted? Are generic types used where useful?

Django

  • Are there any extraneous commas after the field definition? (Something like foo = models.Field(...),, noting the trailing comma, will never be registered with Django as a field.)
  • Do model fields have help_text? Do foreign keys have on_delete and related_name? Are DecimalField defaults actually decimals?
  • Did any foreign keys get added? If so, are they blacklisted on their admin interfaces inside raw_id_fields?

Conclusion

Remember, you are not a machine, and you are supposed to be reviewing code in ways that machines cannot already do.
This requires critical thinking, and critical thinking requires clear thought, an evil mindset, and the ability to think outside the box.

Why You Shouldn't Host Anything for Anyone Else

I know about the 3-2-1 rule of backups (3 copies of the same data in 2 types of media, with one offsite), but never really acted upon it. My usual backups only consist of my most important documents stored in another hard drive, which is, I figure, good enough. With the backup hard drive also aging, I look into different options, like Dropbox, Google Drive, and Syncthing.

The problem with Google Drive is that there is no desktop client available for Linux, and I must remember to back up the data manually, which I do not. What that means is the data that is backed up is neither useful or organised, with no sure way to tell if a document is up to date. If a backup is out of date, what is the point?

Then I give Dropbox a try. Dropbox, on the contrary, does have a Linux client, and they have an online interface that shows history of individual files, which is a nice touch. That, of course, also has its downsides. Every file is accessible by Dropbox itself, and encryption software like Cryptomator are under development, so once encrypted, there is no guarantee that the file can be accessed again.

Finally, Syncthing. Syncthing is the private BitTorrent of sorts, turning your machines into “seeders” for your own files, to be leeched by your own clients elsewhere. Sounds good on paper, but how do you make sure that remote server that you sync with does not expose these files to the filesystem? There is no way to do that; you must have control over your own machines to make this sort of guarantee.

At this point, I seriously consider purchasing a small desktop dedicated for Syncthing. Whenever I change a file on my master laptop, the changes will be stored in that desktop. But hang on, the 3-2-1 rule says this desktop should be elsewhere. where would I put it?

I turn to my own parents. They are not free to do such a thing. I turn to cloud providers. I don’t like the idea of them having backdoors to their own disks (which is justified, in a way). I turn to my colleagues. They said along the lines of:

  • There might be legal implications of me hosting your data
  • There might be legal implications of me hosting illegal data
  • I can’t guarantee the integrity of your backups
  • Having your data on my computer is a legal disaster
  • Having your data, even if you supply your own computer, connected to my home network, is still a legal disaster

Speechless, I turn to yet another colleague, who suggests dumping encrypted backups to S3 or Glacier with a cron job.

But S3 is not free.
But it probably won’t complain about lawyers and stuff.
But you can’t restore from S3 like a human being.
But that’s what it’s like to have two different backup media.

Sigh. Tarsnap it is.

Living Alone: Pros and Cons

My landlord is letting his place go on sale. This also means my eviction is in order (and, given the reason, has a very generous notice required by law). It is time to revisit what it was like to live alone, and what people with similar experiences gained/missed.

Pros

  1. No one will ever touch your food.
  2. The whole fridge is yours.
  3. You can talk to yourself without anyone else thinking you are crazy.
  4. Cleaning schedule is “whenever”.
  5. No quiet hours (except otherwise specified by building).
  6. No one touches your electrical appliances (if any), and they are always available to use. Wash your clothes at 3am if you want.
  7. You can sing in the shower as poorly as your heart desires.
  8. You are almost guaranteed more space per capita.
  9. The door is where you put your shoes.
  10. Thermostat is set by you. Air conditioning is set by you.
  11. Your place may be eligible for 100% of the parking spots (which is usually one).
  12. Can poop with door open. The stench is always your very own.
  13. The bathroom is always free. Toilet paper lasts forever.
  14. Everything is always where you left it.
  15. You can have friends over any time.
  16. You can be completely naked, anywhere, anytime. You can also do anything while naked, anywhere, anytime.
  17. No one uses your soap. You can be sure your razor has not touched the testes of any man (or any other man).
  18. All the Internet bandwidth belongs to you. Also, no one will hack you from the inside.
  19. The sink can be used, perpetually, for soaking dishes that really needed soaking, rather than being called a lazy bastard every other hour.
  20. You can keep various pets and plants with only your rental agreement and/or the law to worry about (where applicable).
  21. You can choose where to sleep. You can sleep on the couch, the laundry room, or above the toilet bowl for all we care.
  22. You can fill the entire house/apartment with cooking smoke and only you will be shamed of what you did.

Cons

  1. Appliances might not come as standard. May need to do your laundry elsewhere.
  2. No noise made by other humans. Can be lonely.
  3. You pay all bills not covered by the lease. This is less likely when living with others.
  4. More expensive rent per capita. Sometimes also higher per unit area.
  5. If anything squeaks and leaks or falls apart, you know you are the only person who could have broken it.
  6. You are almost certainly guaranteed not to have a dishwasher. You might not even have a proper stove.
  7. If anything is dirty, you are cleaning it. Though, you know you were the one dirtying it in the first place.
  8. With no one to gauge your QOL, it is completely possible that you will become a social degenerate.
  9. Inability to share 4L bags of milk.
  10. You never carpool (except if you drive for Uber, maybe).
  11. You never share the cost of your car.
  12. No one will share their Netflix account with you.
  13. If you have a pet, you and you alone must walk it.
  14. You alone will defend the home intruder.
  15. You can invite friends over only if you have the space. You might not have the space.
  16. Losing keys can be a very costly ordeal.
  17. No one to practice interviews with. Or talk to, on a daily basis.
  18. If you torrent anything, they will know it is you.
  19. Too much free space might pressure you to get all sorts of furniture you never needed.
  20. You pay your own home insurance.
  21. Snow? You are shovelling (where applicable). Grass? You might get away with paying a boy to cut it for you, but that also costs money.
  22. It is less likely that you will find someone willing to help you move.
  23. If you are a stroke or heart attack, you die.

Roommates: Red Flags

The community list of red flags when assessing a roommate/potential roommate is as follows:

  • Insists on putting their name on their own food in the fridge, while eating all unlabelled food when no one else is doing it.
  • Has a mother move in for them. Chances are, this roommate will not be doing their own chores.
  • Expects the place to be clean, yet, does not clean anything.
  • Has a smell that lingers wherever they go.
  • Was forced to move out by their former roommates.
  • Is close to you. Once they move in, their flaws magnify. The closer you were, the more flaws you’ll find. Not a bad thing if they are nice though, since that magnifies positively as well.
  • Mentions how clean the place is before/upon move-in. Indicates different living/cleanliness standards.
  • Is a problem drunk. Sleep drunk is their own business.
  • Posts post-it notes in the house. Evidence of passive-aggression.
  • Always keeps the door open (sociopath), or always keeps the door closed (anti-social).
  • Leave dishes lying around.
  • For some reason, showers or poops with the door open, or showers and poops at the same time.
  • Uses metal utensils on non-stick pans (only madmen do this).
  • Has incredibly loud sex, but does not have the courtesy to invite you.
  • Drops weights on hardwood flooring.
  • Is alcoholic, and expects roommates carry them home.
  • Doesn’t have a job, and is not looking for one.
  • Has no sense of humour.
  • Distributes drugs outdoors as an occupation.
  • Has a Tumblr account.
  • Says, “I like to party occasionally.” If they have to say it, they know very well what occasionally means.
  • Is in debt.
  • Is a Republican. Even if your country of residence does not have a Republican party, this still applies.
  • Mentions to you, and makes sure everyone knows, that someone has a restraining order against them.
  • Has a degree in chemistry, and asks if anyone else is using the shed.
  • Smokes, or if you find any indoor ashtrays.
  • Mentions “I cook bad food”. That just means “I burn the stove a lot”.
  • Is the bad kind of racist. “Bad” is the difference between “haha Germans can’t make jokes” and “haha Germans should kill themselves”.
