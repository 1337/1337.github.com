Demystifying the For-Else Loop

The for-else loop is perhaps the worst-named construct in python (not the worst construct, just the worst-named). Nothing about the word else suggests the semantic meaning, “otherwise”. To see what we should name / should have named it, let’s see what it does and how it behaves.

(All experiments run with python3.)

Looping through a list, else…?

>>> for i in [1]:
...     print(i)
... else:
...     print('ha')
... 
1
ha

We didn’t break inside the loop, so this is normal.

Looping through an empty list, else…?

>>> for i in []:
...     print(i)
... else:
...     print('ha')
... 
ha

This time, despite the block having never run, the else block still runs. Then again, we didn’t break in that empty loop either, so this is normal.

Looping through a list, break, else…?

>>> for i in [1]:
...     print(i)
...     break
... else:
...     print('ha')
... 
1

So it appears breaking is really the only thing that makes the difference. But what if the list is empty?

Looping through an empty list, break, else…?

>>> for i in []:
...     print(i)
...     break
... else:
...     print('ha')
... 
ha

It gets more and more reassuring that else is run only if a break statement is reached at any point within the for loop.

Looping through a list, trip an exception, else…?

>>> try:
...     for i in [1]:
...         print(i)
...         raise Exception
...         break
...     else:
...         print('ha')
... except:
...     print('boo')
... 
1
boo

Conclusion

These experiments show that the only way to trigger an else block is when the for loop did not reach any break statements. Therefore, else should be thought of as if not break.

Can You Seed a Random Function With `Random()`?

Let’s find out.

import random


test_set_size = 10000000


def func1():
    eh = []
    for _ in range(test_set_size):
        eh.append(random.randint(0, test_set_size))
    return len(set(eh))


def func2():
    eh = []
    for _ in range(test_set_size):
        ran = random.randint(0, test_set_size)
        eh.append(ran)
        random.seed(ran)
    return len(set(eh))


print(func1())  # 6321602
print(func2())  # 2696

Answer: no, not at all. Don’t ever do it.

Python: Typing Optional

Ah, python typing. In general, I would say typing has made our lives better (than, say, writing types in the docstring, or not documenting types at all), but the Optional type has always confused me.

So, Optional[SomeType] is the exact same as Union[SomeType, None].

But it does not mean a parameter with a default (making it optional) should be annotated with am Optional.

For example, this function does not require an Optional:

def foo(i: int = 1) -> None:
    pass

On the other hand, if None is a valid argument, when Optional is annotated, whether or not the argument is actually optional.

# Yes
def foo(i: Optional[int] = None) -> None:
    ...

# Required, but also yes
def foo(i: Optional[int]) -> None:
    if i is None:
        ...

I guess there is one exception, where a parameter accepts exclusively None (what use is that?). In that case, since having Union[None, None] is stupid, so is Optional[None].

# Yes
def foo(i: None): -> None:
    ...

# No
def foo(i: Optional[None]): -> None:
    ...

  • console.dir(object) // contents may change as code executes
  • console.group() // starts a group (groupCollapsed starts a collapsed group)
  • console.groupEnd() // ends a group
  • console.table(objOfObjs, listOfKeys) // prints out a table of these things
