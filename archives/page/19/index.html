PEP8 for the Working Class

Do you think Python? Congratulations! You are in for a world of work. Work means that, in order to upset fewer of your colleagues, you have to follow a set of rules that is not present in the current PEP8 (or the previous iteration, for that matter.) This post prepares you to become the world’s most compliant python (at this time, python2…) developer for times to come.

Definitions

The key words “MUST”, “MUST NOT”, “REQUIRED”, “SHALL”, “SHALL NOT”, “SHOULD”, “SHOULD NOT”, “RECOMMENDED”, “MAY”, and “OPTIONAL” in this document are to be interpreted as described in RFC 2119.

Arguments

Builds SHOULD NOT pass if:

  • any *args are used other than package compatibility, and
  • any **kwargs are used other than package compatibility.

Comments

Builds MUST fail if:

  • The code-to-comment (by line count) ratio is less than 1:3;
  • The first line of every character of non-block comments is not in upper case.

Dictionaries

dicts:

  • MUST have keys of type str. Not unicode, not anything else; str.

  • Division MUST take place with both sides of the operator as:
    • float
    • decimal.Decimal
    • something more precise than decimal.Decimal

Docstrings

Docstrings MUST be of reStructuredText Docstring Format.
Docstrings MUST fail a build if:

  • They do not describe what is happening;
  • They describe incorrect parameters;
  • They describe parameter(s) incorrectly.

Docstrings MUST:

  • Be under 72 characters long per line;
  • Have a blank second line;
  • Declare the return type; and
  • Declare the exceptions returned (None if never).

Functions and Methods

  • Function name SHOULD NOT be under 5 characters long, and MUST NOT be under three characters long.
    This is largely due to val() being a sensible function name.

  • Functions and methods MUST:

    • Be under 100 lines long.
    • Have fewer than 5 return statements.
    • Accept fewer than 5 parameters. *args and **kwargs each count as one parameter.

  • Builds MUST fail if a method’s signature is different from its parent method’s signature.

  • Functions SHOULD always check its signature and raise AssertionError if the parameters are not of spec.

Ignores

  • There MUST NOT be any ignore statements that skip this document for any
    project files (excluding third-party packages).

Imports

  • Imports MUST use the import x.y.z notation.
  • Builds MUST fail if an imported module is renamed: import x as y
  • All imports MUST be included in a separate line. No from x import y, z.
  • Builds MUST fail if the from x.y import x notation is used.
  • Builds MUST fail if the imports are ordered in any way, other than alphabetically, using the first character after import.

Lines and Statements

A statement MUST contain only one statement.

Each line MUST be under 72 characters long – because Python docstrings
are 72 characters long. Code lengths are understandably 79 characters long,
but if you are going to set a limit on docstrings anyway, you might as well
limit the code to 72 characters long also.

Introduction to HTML5 and Accessibility (2015)

The Importance of Accessibility

I’ll save myself some time writing up things that others that already have:

Accessible Web pages benefit everyone. The more accessible your content is, the more user-friendly it will be to all people, whether reading the page on a large screen or on a cell phone, or hearing the page read aloud by a screen-reader.

Principles of Accessibility

At the top are four principles that provide the foundation for Web accessibility:

  • Perceivable: the information must be available to at least one of the user’s senses.
  • Operable: the interface cannot require interaction that a user cannot perform.
  • Understandable: dumb it down.
  • Robust: design the content to be read by present and future (assistive) technologies.

Types of Accessibility to Cover

  • Visual disabilities (e.g., blindness, low vision and lack of color perception)
  • Hearing disabilities (e.g., hard of hearing, deafness)
  • People with physical disabilities (e.g., limited strength, reach or manipulation, tremor, lack of sensation)
  • People with speech disabilities
  • People with language, learning or cognitive disabilities (e.g., reading disabilities, thinking, remembering, sequencing disabilities)
  • Other disabilities (e.g., epilepsy, short stature)
  • Individuals with any combination of these disabling conditions (e.g., deaf-blindness)

Considerations for the Visually Impaired

People who are visually impaired often rely on screen-readers to read Web pages. Screen-readers translate Web pages into plain text, which is either read aloud by a voice synthesizer or translated by a Braille device…

Optimising for Screen Readers

In general,

  • Avoid “click here” links or vague link names.
  • Use semantic headings (H1, H2, etc.).
    “The H1 heading is in the banner”
    “The H2 heading is the title of the article”
    “So, in the body of your article, use H3 through H6 for your section heads.”
  • Use punctuation.
  • Keep words and layouts simple; only include words that need to be read aloud.
  • Images must have an alt attribute that makes sense.

Low Vision and Colour-Blindness

  • Ensure page can be zoomed, even on a mobile device.
    • WCAG requires to be > resized without assistive technology up to 200 percent in a way that does not require the user to scroll horizontally to read a line of text on a full-screen window.
  • Text and images must have sufficient contrast.
    “WCAG 2.0 Level AAA requires a contrast ratio of 7:1 for normal text and 4.5:1 for large text.”
    • BUT WCAG advises “using a light pastel background rather than a white background behind black text to create sufficient but not extreme contrast”
  • Empty text fields should not have the same colour as its parent.
  • Don’t rely on color alone to convey information.

Considerations for the Hearing Impaired

Some people with auditory disabilities can hear sounds but sometimes not sufficiently to understand all speech, especially when there is background noise. This includes people using hearing aids or other approaches to improve the sound.

  • Use video captions. Use <caption> tags.
  • Allow volume to be adjusted for videos.
  • Make as few content dependent on hearing as possible.
  • Provide sign language videos for audio-only files.

Considerations for the Physically Impaired

Physical impairment doesn’t just mean not having arms.

Physical disabilities include weakness, involuntary movements, tremors, lack of coordination, paralysis, limitations of sensation, joint problems (such as arthritis), …

  • Provide keyboard shortcuts.
    • Keybaord shortcuts cannot be combinations that require timing.
  • Ensure tab index is proper, and that all interactive elements can be focused.
  • MUST NOT impose a time limit to fill out a form or click a button.
  • MUST NOT rely on hover or complex navigation mechanisms.

Considerations for Speech Impairment

  • Avoid microphones.
  • Offer text input for situations where a microphone is necessary.

Considerations for Cognitive Impairment

… can impact how well people hear, move, see, speak, and understand information, …, not necessarily intelligence…

So it doesn’t just mean dumbing down the information. The problems that you can
eliminate mainly concern complex operations that are simply unnecessary.

  • Avoid complex navigation mechanisms.
  • Avoid long sentences that require the reader to keep track of its contents.
    Use this tool to check your language comprehension.
  • MUST NOT make objects blink, flicker, or move too quickly.
  • MUST NOT play background audio.
  • Offer stop buttons for music and videos.
  • To achieve WCAG 2.0 AAA compliance:
    • An expired session must be resumed completely after the user reauthenticates.
    • Breadcrumbs will be required if there isn’t a sitemap.

When and How to Use Which Tag

<*>

  • Use readable fonts. Check with the Colour contrast checker to make sure text is readable.
  • Text sizes must be in either %, em, or one of the named sizes.
    • line-height must also be defined with a percentage between 150% and 200%.
  • Any element that has a role must have an role attribute.
  • You should never define these abstract roles on any element:
    • command
    • composite
    • input
    • landmark
    • range
    • roletype
    • section
    • sectionhead
    • select
    • structure
    • widget
    • window
  • Text should not be justify aligned.
  • Text should not be center aligned either.
  • Columns should have limited width.

<a>

  • Links must have a :focus style that is different from its normal style.
  • Links should always be underlined by default.
  • Links must define role="link", except when the link doesn’t navigate anywhere, in which case its role is button instead.

<abbr>

  • If a <dfn> defines an acronym or an abbreviation, this needs to be in <dfn>.
  • <acronym> is non-standard so there is no debate on which one to use.

<b>

  • MUST NOT use <b>.
    Consider using <strong>, or if the style has no semantic meaning, use a bold CSS style instead.

<body>

  • [Always declare a background-color.
    Mostly because it is near-free to declare a background colour for no drawbacks.

<div>

<div> is the single most abused element in the HTML spec, primarily because
it has no meaning.

  • <div> should contain lang and title if it is not enclosed by one that does.

<dfn>

  • <dfn> should always contain an ID for later reference.

<form> series: <button>, <fieldset>, <input>, <label>, <legend>, <select>, <textarea>, …

  • <form> must have a role, e.g. <form role="search">.
  • For interactive elements, aria-disabled must be defined, and either true or false.

  • If an element is only visible after some user action, authors must set the aria-hidden attribute to true.

  • Besides invalid and ng-invalid, aria-invalid is also required, and be either true or false.
  • Elements that have readonly defined should also define aria-readonly.
  • <input type="text"> also has an role="textbox".
  • Optional fields should be hidden.
  • To get WCAG 2.0 AAA, submissions must be reversible, and have a mechanism to confirm.
  • Forms should not have a reset button.
    “The worst problem about Reset is that users click the button by mistake when they wanted to click Submit . Bang — all your work is gone!”
    • “Exception: Use Reset for Repeated Form-Filling” (RTFA)

Sliders

  • Sliders must define all these attributes at once:

<heading>, <header>

  • There is no <heading>.
  • Headings (<h1>, <h2>, <h3>, <h4>, <h5>, <h6>) must come in that order.
  • <header> must be used to wrap the site’s primary <nav>.

<hgroup>

  • You MUST NOT use <hgroup>.

<html>

  • <html> requires a lang attribute (it is not always en, by the way)

<i>

  • Unless the tag encloses a term in a foreign langage, MUST NOT use <i>.
    Consider using <em>, or if the style has no semantic meaning, use an italic (or cursive) CSS style instead.
  • Language terms must have a lang attribute, e.g. <i lang="fr">Merde</i>.
  • Language terms that are in the page’s language should consider <dfn> instead.

Font awesome

Using <i> for fontawesome icons is in fact invalid WCAG 2.0 code.
Simply use <span> in place of <i>.

<img>

  • <img> is self-closing in HTML.
  • <img> tags require a unique and meaningful alt attribute.

<label>

  • In addition to a matching <label>, all form elements (except <label>, of course) must define either an aria-label or aria-labelledby attribute.
  • If the label text is visible on screen, authors SHOULD use aria-labelledby and SHOULD NOT use aria-label.
    • And if the label is long, use aria-describedby instead.

<link> and <meta>

All pages should contain these meta tags for meaningful accessibility and SEO.
More info about meta tags

Optionally, pages can have these tags where applicable:

Open graph tags

While OG tags MUST NOT do anything to aid in accessibility, these commonly-used
OG tags (not mentioned above) should be added anyway:

<nav>

  • All elements in a navbar must be enclosed in a <nav>, despite that link saying it is optional.
  • <nav> must have role="navigation":
    • First level <ul>s must have role="menubar"
    • Second level <ul>s must have role="menu"
    • Popups must have role="presentation", while the trigger must have aria-haspopup="true".

<option>

  • aria-selected and aria-checked are optional attributes that override focus state.

<p>

  • Semantically, only when it is a paragraph.
    Do not use <p> for padding or spacing.

<script>

  • Never write inline scripts.
  • /* <![CDATA[ */ and /* ]]> */ are necessarily only when the doctype is XHTML.

<select>

  • Is there an <optgroup> where applicable?

<span>

  • Just because you can use spans for everything, it doesn’t mean you should.
    Prefer semantic tags over <span> where possible.

Tooltips

A tooltip whose role is tooltip should be loaded before any <input>
uses that tooltip in its aria-describedby attribute.

<strike>

  • Never use it.

<textarea>

  • Text areas have an role="textbox".
  • aria-multiline must be true.

<u>

  • You MUST NOT use <u>. Use text-decoration: underline instead.

Python Software Quality Checklist

You either commit yourself as a professional racing driver that’s designed to win races or you come second or you come third or fifth and am not design to come third, fourth or fifth. I race to win. And if you no longer go for a gap that exists, you are no longer a racing driver because we are competing, we are competing to win. ― Ayrton Senna

Shit code is not code. Shit code is shit.

Databases

  • Are all database calls resistant to race conditions?
  • For transactional queries, are they definitely not idempotent
    (e.g. you don’t want to subtract value and still read the same balance again)?

Docstrings and Comments

  • Are all files documented?
  • Are all function and method parameters documented?
  • Are all lines under 72 characters long?

Exceptions

  • Does the code fail early?
    • And is there anything catching all of these exceptions?
  • Are all custom exceptions based on Exception?
  • Do these exceptions handle unicode characters?
    • Messages are empty if the string fails to encode.

Files

  • Are all lines under 79 characters long?
  • Are any files too long? (pylint checks this.)
  • Are any files organised inappropriately?
  • If it is an executable file, is there a shebang (#!/usr/bin/env python...)?
    • Should it be executable?
    • Does the shebang point to something relative?
    • Does the file have executable permissions?
  • Does it declare an encoding?
  • Does it have a docstring? See Docstrings and comments

Forward Compatibility

  • Is there anything that prevents migration to python3?\
  • Are classes for supporting compatibility named with the 3K notation?
    • Python 2.7: python_2700
    • Python 3.0: python_3000
    • Python 3.4: python_3400

Functions and Methods

  • Is the code reinventing the wheel? Are there existing functions that do the same thing?
  • Does the function use keyword and star-args unnecessarily?
  • Can it be a pure function? See Tests
  • If it returns just a boolean, does the function have a name that appropriately reflects that?
  • Do number parameters have min/max/allowed_values check(s) in place?
  • Are there any magic strings, numbers, and/or values?
  • Are all function arguments given with keywords where appropriate?
  • Do your functions survive the grep test?
    If you cannot grep a function by name and get all its usages, it fails the grep test.

Function lengths

“The first rule of functions is that they should be small. The second rule of funct`ions is that they should be smaller than that. Functions should not be 100 lines long. Functions should hardly ever be 20 lines long.” - Bob Martin, Clean Code

“Every function was just 2, or 3, or 4 lines long, each was transparently obvious, each told a story, and each led you to the next in a compelling order.” - Kent Beck

NetBeans heavy recommends a 10-line function length to keep them testable.
Others say 10 lines is too short and should be under 8, 20, 60, or 53.
The takeaway is: keep it under 7 to be compliant with all of these recommendations.

Imports

  • Are all imports ordered?
  • Besides __future__, are there from x import y line(s) x.y could have been used instead?
  • Do __init__.py files declare a __all__?

Iterables

  • Are there any instances where iterables are unnecessarily lists instead of generators?
  • Are there any usages of the deprecated sets library instead of set/frozenset?
  • Are there any instances where frozenset could have been used in place of set?
  • Are all iterables iterated using enumerate(), iteritems(), or itervalues()?
  • Are there instances of not x in y instead of the better x not in y? (pylint checks this.)

Logging

  • Are all logging calls important?
  • Is the string formatted by logging? logging.info(u'%s', 'foo')

Migrations

  • Are migration files missing?

Numbers

  • Is there a from __future__ import division import?
    • If not, are there instances of division where one of the numbers can be an integer?
  • Does code that perform float(some_string) also catch UnicodeEncodeError?

Regex

  • Is it necessary?
  • Does any of them contain unnecessary and unsafe .?

Speed

  • Write x < y < z instead of x < y and y < z.
  • Are there any database queries occurring unnecessarily in loops?
  • Are there instances of while True where while 1 can be used instead?
  • Are there any for loops that could have been faster list/dict/generator comprehensions?
  • Is the code “perceivably fast”?

Statements

  • Do lines contain more than one statement?
  • The We Are All Consenting Adults Principle: are you or anyone in the team a child? Do you know for sure? Would you risk it?

Strings

  • Adapting this person’s practice, does this double-quoted string contain variables?
  • If it is visible to the user, is it a unicode instance?
  • Besides logging, are all strings formatted using .format() instead of %?
  • Are all format() calls made using keyword arguments?

Templating

  • Are there untranslated strings?
  • Are there hard-coded values?
  • Is the template coupled to what provides its variables?

Tests

  • Is there at least one test case for every pure function?
  • Are most (if not all) code paths tested? This can be detected through coverage.
  • Is there a test involving unexpected input types?
  • Did you test your changes locally, no matter `how small, before re/submitting a change?
  • Did you test locally, again, after a rebase?
  • Tests with conditional statements (e.g. for, if, with, while) MUST double-check if it is testing anything.

Variables

  • Are there unused variables? (pylint checks this.)

When to Use Unicode Strings

Still stuck on Python 2.x? Bad news, strs are not unicodes. That means you
will need to know when to use unicode and when to handle str. Should you
use unicode? When should you use it? Are there gotchas? Let’s find out.

What is a Python unicode?

Let’s first establish what a str is.

strs are… strings of bytes. They have series of 1 and 0, encoded in
a format defined when the string was generated.

unicode, on the other hand, abstracts the concept of bytes away from the
programmer, so byte-counting is no longer necessary.

Encoding

Since strs are just series of bytes, there is no limit to what they can hold.
A string can contain “unicode characters” as we understand it, because “unicode
characters” are, also, series of bytes.

What is str.encode()?

str.encode() does nothing.

What is unicode.encode()?

unicode.encode() always returns a str.

Decoding

What is str.decode()?

str.decode() always returns a unicode.

What is unicode.decode()?

unicode.decode() does nothing (if the encoding is utf-8).

Why should strings be encoded and decoded?

It is important to decode your str to a more specialised instance as soon
as you got it, because unicode counts characters properly. For example, in
this example, "Krstić" appears to have six
characters, right? But since that is just a series of bytes:

Implicit conversion

Mixing str and unicode automatically decodes the str to unicode
using the default system encoding, which is usually 'ascii'.

This means decoding errors can occur if the string is not encoded in UTF-8.

When to use unicode

tl;dr Almost Everywhere. With the exception of files, just remember:

  • decode (to unicode) early
  • use unicode where the text is meant to be displayed
  • encode (to str) only when necessary

ASCII strings

YES! With the exception of internal strings (e.g. keys), there are no
drawbacks to using unicode strings for storing ASCII characters.

A caveat to doing this is if there is any poorly-written code that explicitly
checks if isinstance(your_string, str), in which case, using unicode is
not an option, unfortunately.

It is not advisable to compare str instances with unicode instances
with the == operator. The example there is just for reference.

Dictionary keys

Similarly, any ASCII dictionary keys are directly accessible with its unicode
accessor, and vice versa, but unless your dict keys will be seen by the user
(which is possible in some cases), you should not be doing it.

Implementation

Although there is a bytes thing in python2, it was not implemented
properly. Python3 implemented it properly, but not for python2.
What this means is although bytestrings should be used for dict keys in
python3, this does not concern python2.

somepackage.some_function(str)

If a function says it takes a str, pass a str. Otherwise, hope for the best
and assume duck typing works on the unicode.

If you pass unicode into a function you do not control, be ready to
“be in for a world of hurt“.

Docstrings

Where necessary,
docstrings can be implemented with the u""" syntax """. However, there is no
substantial merit to doing so.

Logging

There is no difference, it seems, between logging strs and unicodes,
even when the unicode string is %s‘d into a standard string.

Printing

The fact that a str can contain unicode characters confuses me, but the
rule of thumb is: if someone is going to read this text, it should be unicode.

Anything calling unicode(variable)

Watch out when using a function, or a package function, that explicitly calls
the unicode() function, because passing unicode into unicode() fails
ironically.

This is because unicode() accepts a str as its first argument. To confuse
us further, unicode() actually accepts unicode objects that are purely
ASCII:

Reading and writing to a file

When files are concerned, using str (remember, series of bytes) is much more
intuitive than using unicode, which can lead to undesirable errors.

__main__

You can use unicode for __main__, but you shouldn’t; this is largely due
to the fact that there are no unicode characters in "__main__" to begin with,
and that __main__ is not user-facing text.

Python’s file() selected 'ascii' because python2’s default encoding is
'ascii', making it impossible to write some unicode instances directly.

from __future__ import unicode_literals

unicode_literals changes all instances of 'strings' in a module to
type unicode. This seems harmless at first, but it also changes the return type
of all string functions in a module, making it hard to debug.

Avoid unicode_literals. This is also the explicit unicode literals PEP
passed, reintroducing u'unicode literals' back in Python 3.3+.

To handle unicode in projects that support both python2 and python3, investigate
the use of six:

When using Django

Django 1.5+ provides various helper functions for handling unicode in your
project. They include @python_2_unicode_compatible,
which automatically adds __unicode__ to any model with __str__, and
smart_text (lazy) / force_text (non-lazy), which guarantee to return
unicode instances.

six is installed as a dependency.

When a str is not necessarily encoded in UTF-8

Just because UTF-8 gets all the love, it does not mean all strings are encoded
in it. For example, PHP defaults to UTF-16, which would be wrong to decode
to a UTF-8 unicode instance. To guess the encoding of something, use
chardet.detect().

Again, decode as early as possible. Passing a string of unknown encoding out
of the data model will inevitably cause problems when it needs to be spliced.

When UTF-32 is used

Python2 does not support UTF-32.

  12. You, as a new member at StartupCompany, are very special. We treat you like a team. The whole team.
  13. We use Dipchat as our company issue tracker. If you find a bug, please file it inside the room named ‘Developers’. The bugs that you file will not be resolved; the bugs that you address will not be acknowledged.
  14. The Business team has no technical knowledge related to building and designing a product. As such, if they ask you how long something would take to implement, please work with them and reply with ‘a very short time.’
  15. We are a six year-old company, and we have made a lot of great products in the past. You will be working on a relatively new project that needs to have features from all previous iterations.
  16. After reading about Exception-driven development (EDD) on Hacker News, we have adopted it as a practice. The moral of the story here is that if we monitor our systems, then they will not fail.
