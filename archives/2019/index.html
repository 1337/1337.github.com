Commenting

Excerpts. Interpretation of the source. If you want to see examples for any of these, go check out the source.

Why write comments

Comments are rubber duck debugging on steroids, except you are not talking with a rubber duck, but with the future reader of the code. You explain why what you are doing is acceptable, honorable, or good enough. And if it is not, you make your homework, and come up with something more decent.

Comments also help others understand you and your work.

Do write: Function comments aka docstrings

Function comments are actually a form of in-line API documentation.

  • The reader should be able to use a function without understanding the implementation, i.e. treating the function as a black box.
  • Putting docstrings for functions directly where they are defined helps the reader focus on the code, instead of context switching between code and documentation.

Notes:

  • You need to update the docstring whenever you change the code. Yes you need to check this when you change the function, but the benefits outweigh the “the code is the only source of truth” argument.

Do write: Design comments aka module docstrings

The design comment states how and why a given piece of code uses certain algorithms, techniques, tricks, and implementation. It is an higher level overview of what you’ll see implemented in the code. With such background, reading the code will be simpler.

  • Why is this trivial-looking module not actually trivial?
  • Why is a particular algorithm design chosen instead of something else? (explains why the author is not just being lazy.)

Notes:

  • Apart from function and design comments, other types of comments here don’t necessarily come in the form of a docstring.

Do write: Why comments

Why comments explain the reason why the code is doing something, even if what the code is doing is crystal clear.

  • Why is this function clearing the user cookie?
  • Why is this function clearing the user cookie after a certain function call, and not before?
  • Why is this thing written this way, and not a more obvious way?
  • Why shouldn’t I change this line here? (hint: it’s harmful for reasons that are not obvious.)

Do write: Teacher comments

Teacher comments teach the reader the domain (for example math, computer graphics, networking, statistics, complex data structures) in which the code is operating, that may be one outside of the reader skills set, or is simply too full of details to recall all them from memory.

  • What is the haversine formula?
  • What is this data structure that the code is using?
  • Corner cases that this implementation is, and is not, designed to take care of.

Notes:

  • Teacher comments do not relate to the code.

Do write: Checklist comments

Checklist comments tell you if you do one thing, you need to do other specific things as well.

  • It tells you a set of actions to do when something is modified.
  • It warns you about the way certain changes should be operated.

Notes:

  • Unlike “why comments”, the checklist comment used in the same case is more biased towards telling you what rules to follow if you want to modify it, without breaking the code behavior.
  • Checklist comments are very common inside the Linux kernel, where the order of certain operations is extremely important.

Do write: Guide comments

Guide comments’ sole reason to exist is to lower the cognitive load of the programmer reading some code. They are (…) what most people believe to be completely useless comments.

  • Guide comments are step-by-step, line-by-line comments that make a piece of code easier to read.
  • They’ll acknowledge you about the fact you are understanding it right.

Notes:

  • If the code isn’t actually hard to read, these comments become “Trivial comments” (see below). If you want to leave guide comments, make sure the comment is necessary.

Do not write: Trivial comments

  • Trivial comments tell you, say, it is incrementing a variable, on the line that increments a variable.
  • Trivial comments state “what the code is doing, and not why.”

Do not write: Debt comments

FIXME, TODO, XXX, “This is a hack”, are all forms of debt comments.

The issue with debt comments are these FIXMEs might require a different kind of fix in the future, which may mislead a future author into doing the wrong thing.

Do not write: Backup comments

Backup comments are commented-out code.

Why Is Gpg So Hard?

Can’t we learn just the basics first?

Managing keys

Your own keys

  • List the keys you have: gpg --list-secret-keys
  • Create a new key: gpg --gen-key
  • Renew an existing key: gpg --edit-key <key-uid>, then type in expire, which also allows renewal.
  • Set the default key: edit default-key <key-uid> in ~/.gnupg/gpg.conf
  • Publish your key to the key server: gpg --keyserver pgp.mit.edu --send-keys <key-uid>
  • Revoke a subkey: gpg --edit-key admin@example.com, then type revkey on a subkey. Finally, publish your key to the key server (see above). In general you try not to use or revoke your master keys because of this.

Other people’s keys

  • List all keys: gpg --list-keys
  • Search for someone’s key: gpg --search-keys admin@example.com, which also prompts if you want to import it.
  • Import a key from the key server: gpg --recv-keys <key-uid>
  • Import a key from a file: gpg --import key.asc, with “.asc” being ASCII-armoured
  • Trust a key: gpg --edit-key admin@example.com, and then type trust.

Using keys

Encryption

  • Create a master encryption key: gpg --gen-key, then select (1) RSA and RSA (default), which both encrypts and signs.
  • Create a encryption subkey: gpg --edit-key admin@example.com, then type addkey. You will be given one or more (encrypt only) options.
  • Encrypt some text: echo 'text' | gpg -ear admin@example.com
  • Decrypt some text: gpg --decrypt, paste in the armoured text you got from the command above, then press Ctrl+D.
  • Encrypt a file with a recipient: gpg -er lpppppl@gmail.com file (makes file.gpg)
  • Encrypt a file with a passphrase: gpg -c file (makes file.gpg)
  • Decrypt a file: gpg -d file.gpg > file

Signing

Signing also timestamps a document, so not only do you know who signed it, but also when they did, and if they resign something later, it is going to show.

  • Create a master signing key: gpg --gen-key, then select either (1) RSA and RSA (default), which both encrypts and signs, or (3) DSA (sign only) or (4) RSA (sign only), which only sign.
  • Create a signing subkey: gpg --edit-key admin@example.com, then type addkey. You will be given one or more (sign only) options.
  • Sign some text: echo "Text" | gpg -s > OutFile
  • Sign a file: gpg --armor --detach-sig --sign FileName or gpg -abs FileName (without --detach-sig, the output includes the file, which can be huge.)
  • Verify signature on some text: dunno 😂
  • Verify signature on a file: gpg --verify FileName.asc FileName

A Big Request

I wanted to make a reporting resource on a web server. (Some people call me a web developer, but I’m not the best.) Naturally, since reporting should not have any side effects, the GET method ought to do the job.

The GET method is said to serve this purpose:

The GET method requests transfer of a current selected representation
for the target resource. GET is the primary mechanism of information
retrieval and the focus of almost all performance optimizations.
…

What did they mean by primary mechanism of information retrieval? Is there a secondary mechanism of information retrieval? Looking at the rest of RFC7231, they may be referring exclusively to HEAD:

A user agent can perform a retrieval request targeting that URI
(a GET or HEAD request if using HTTP)

So, if you are an HTTP client, you can only use GET or HEAD to retrieve information from a web server. Got it.

But what if your GET request has too many parameters (HTTP 414)?

A big request

I am making GET requests that have too many query parameters. I can see a few potential solutions:

  • Send fewer query parameters,
  • Set up the web server to take more query parameters,
  • Change the GET to some other verb, or
  • Something else.

Solution 1: send fewer query parameters

The first solution is often the most straightforward: if you want to get around “URI Too Long”, then send fewer query parameters. How hard is that?

Well, it is quite hard, as it turns out. Some APIs really to accept requests like site/reports/{filter1:<filter1>, filter2:<filter2>, ...}, and there is no way to shorten these filters.

Conclusion: not feasible

Solution 2: set up the web server to take more query parameters

You can configure Apache to take a longer URI by modifying the LimitRequestLine value, from 8kB to something else, but it also makes the service susceptible to DoS attacks. Also, just because your API accepts longer query strings, it doesn’t mean your client’s browser can send them. “The limit is in MSIE and Safari about 2KB, in Opera about 4KB and in Firefox about 8KB. We may thus assume that 8KB is the maximum possible length and that 2KB is a more affordable length to rely on at the server side and that 255 bytes is the safest length to assume that the entire URL will come in. If the limit is exceeded in either the browser or the server, most will just truncate the characters outside the limit without any warning.”

And we know we can’t force the client to use a particular browser.

Conclusion: not feasible

Solution 3: change the GET to some other verb

Recall that RFC7231 says only GET and HEAD can be used to retrieve information from the server.

So what if we change the GET request to a HEAD?

The HEAD method is identical to GET, except a HEAD request can’t even have a response body:

The HEAD method is identical to GET except that the server MUST NOT send a message body in the response (i.e., the response terminates at the end of the header section).”

The HEAD method does not allow the browser to send more query parameters.

Conclusion: not feasible

Solution 4: Use POST, with X-HTTP-Method-Override

A StackOverflow answer suggests sending the request as a POST (where the parameters are in the request body instead of the URI), with a X-HTTP-Method-Override: GET header, so the web server changes the request back to a GET. Google Translate does that with the text people type in to translate.

The twist: X-HTTP-Method-Override is deprecated. The server also needs to manually sense that override, so it may well be a simple POST request.

While a lot of people have accepted the practice that a GET with too long or too complex a query string (e.g. query strings don’t handle nested data easily) can be sent as a POST instead, many people have not. These people include those who review my code, and this blog, who would rather die use a WebDAV verb called REPORT than POST (“its semantics are well defined and it works everywhere”).

Conclusion: not feasible (at least from a code review perspective)

Solution 5: Use WebDAV’s REPORT method

Since this blog really hated Dropbox changing from GET to POST for similar reasons (“Don’t break the Web!” says the blog), sending query parameters as POST, and getting responses back, is right out. Django does not appear to support a REPORT verb, so that’s right out, too. (Even if it did, how much browser testing would you need to do to give you a required level of confidence?)

Conclusion: not feasible

But… hold on, what did that blog post say?

Solution 6: POST the query, but GET the results

That post says (paraphrasing):

  1. Create a site/reports/search endpoint where you allow clients to submit POST requests, with request bodies containing {filter1:<filter1>, filter2:<filter2>, ...}.
  2. This POST returns a URL (in the body) that tells you where to GET your results, e.g. site/reports/search/88b413ec-fd2f-4269-983e-205eb2566c48.
  3. You GET /site/reports/search/88b413ec-fd2f-4269-983e-205eb2566c48 to get your search results.

This StackOverflow answer also suggests a similar POST /site/reports/search resource, whose sole purpose is to let users submit a search.

The reason this design works is twofold:

  1. The POST request is actually creating a resource, namely a search. (Consider returning a 200 instead of 201, though.)
  2. The GET URI is now short.
  3. Bonus: “The results are still cacheable, safe and idempotent.” (I’m not so sure about that)

Of course, this comes with some complexity not required by a simple GET request. You need somewhere to save these searches. Like a database. If you save it to a database, then the search results may be stale. If you don’t save it to a database (like if you use Redis instead, for example), then you need to worry about users GETting a missing search, which needs to return an HTTP 404 or 410.

Conclusion: feasible, if you aren’t lazy

Conclusion

  • Use solution 4 if you are lazy, or
  • Use solution 6 if people review your code.

How to Install Node.js

tl;dr: don’t install nodejs.

Here at Maymay Research Institute (MRI), we make things with JavaScript.

To run JavaScript outside a browser, chances are you will need something called Node.js.

A change is made to Node.js every other day, so as are the binaries. We joke that node and npm should simply be named after the unix timestamp.

A: we should just version js packages by unix timestamp, because 
   there's a new one every other second
B: That would solve it.
A: that would actually work
A: `npm@1550873940.0.0  // because every change is breaking anyway, this is semver-compliant`
B: Oh my

While that was largely a joke, it is not far from the truth, and having so many versions of node and npm can cause problems on developer machines, namely this code review comment (paraphrased):

There are packages that are built for each particular node distribution (node-sass for example), so when you and I (run) “yarn or npm install” (it will install) different binaries of the same package. … This leads to “On my computer (it) works” problems. (We) should manually enforce the version, and (upgrade) every once in a while if a new LTS version is available (…).

The comment highlights the importance for all developers to run the exact same version of node and npm for a project. But how do we achieve that?

Docker

You can’t just throw everything at Docker, manperson.

nvm use ${specific version}

The team can pin node to a specific version (and therefore the version of npm, as well). They do so by running

nvm use v10.15.2

Normally I would call it a day, since v10 would have been good enough for me, but since “This leads to “On my computer (it) works” problems”, we are going to have to pin it down.

Where do these node versions come from?

Nodejs has its own repository on GitHub, but they don’t provide binaries there. Instead, according to the command that nvm runs:

bash -c "NVM_DIR=$HOME/.nvm && source $NVM_DIR/nvm.sh  && NVM_NODEJS_ORG_MIRROR=https://nodejs.org/dist nvm install vx.y.z && nvm use vx.y.z && export > env.txt"

The binaries come from https://nodejs.org/dist/. In particular, these are the directories we are concerned about:

https://nodejs.org/dist/v10.15.1/
https://nodejs.org/dist/v10.15.2/
https://nodejs.org/dist/v10.15.3/

And they all have SHA 256 checksums for everything you download (assuming you verify them).

https://nodejs.org/dist/v10.15.1/SHASUMS256.txt
https://nodejs.org/dist/v10.15.2/SHASUMS256.txt
https://nodejs.org/dist/v10.15.3/SHASUMS256.txt

That’s all well and good. Having SHA 256 allows team members to verify that we downloaded the same version of node from the repository. But wait…

How do you know their v10.15.2 is GitHub’s v10.15.2?

Surely anyone who hacked https://nodejs.org/ could have replaced https://nodejs.org/dist/v10.15.2/SHASUMS256.txt, https://nodejs.org/dist/v10.15.2/SHASUMS256.txt.asc, https://nodejs.org/dist/v10.15.2/SHASUMS256.txt.sig, and https://nodejs.org/dist/v10.15.2/node-v10.15.2-linux-arm64.tar.gz so you download a hijacked binary. Even if you try to verify the GPG signature with the key displayed in the repository, anyone working for nodejs could have compiled the same version twice. If that happens, you and I may be running different node binaries that both declare themselves to be v10.15.2.

How do you know you are running GitHub’s v10.15.2?

You download from https://nodejs.org/dist/v10.15.2/; it does not mean what you download corresponds to https://github.com/nodejs/node/tree/v10.15.2. The reasoning is twofold. First, anyone who provides node binaries can compile any code with the version changed to v10.15.2. So you have to trust them. Second, the commit that made v10.15.2 was not signed, whereas the ones for v10.15.1 and v10.15.3 are signed with valid GPG keys. This means, no matter how small the probability is, v10.15.2 can be moved without anyone knowing, and projects that use version v10.15.2 can potentially get a binary that is different eventually.

What if everyone compiles their own v10.15.3, then?

(v10.15.3 is a signed commit.) Well that’s not good enough either. BethGriggs can move v10.15.3 and sign the commit again, resulting in a signed v10.15.3 tag that points to a different commit.

But if you pin your node version to a signed commit hash (1a96d83a223ff9f05f7d942fb84440d323f7b596), clone it, and then compile it yourself, you can be sure everyone on your team gets the same v10.15.3 binary, then?

Guess what! The answer is no! You can mess with compiler flags!

What works?

One person on the team should be responsible for compiling node and npm binaries. Then that person signs these binaries and publishes both the binaries and signatures to an internal repository.

Afterwards, all other team members install nodejs using the command

bash -c "NVM_DIR=$HOME/.nvm && source $NVM_DIR/nvm.sh  && NVM_NODEJS_ORG_MIRROR=(your team repository) nvm install vx.y.z && nvm use vx.y.z && export > env.txt"

When you and your team members all install signed binaries from a known source, you can all be sure you have the same binary.

It doesn’t mean the binaries are trustworthy, though. Your team member could have spiked the source with special source before compiling it. That requires you to compile the binary yourself, and verify that the build is reproducible on your end. Nodejs builds are not reproducible, as far as I know.

In conclusion: nothing works. There is always something wrong. Your teammates don’t trust you. Become a farmer. Live off the grid. Lawyer up.
