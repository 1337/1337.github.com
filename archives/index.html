Thoughts on Jetpack Compose

For whatever reason, I have been toying around with Jetpack Compose for about a week now. Having started Android development (out of necessity) however many man-hours ago, I was using Activities to set up my views, and layout files to define my templates. This meant each activity I had must hook up template events one by one, like so:

I looked for a replacement for this style of development from the start. Firstly, that is in Java, and I have never liked Java. Secondly, this is a stupid amount of code for what amounts to $('button').click(() => ...) in something like jQuery. The fact that I need to inflate (show/bind, I guess?) my own layout file, or am given the opportunity to do it differently, means that there is too much flexibility for making a button do stuff. The fact that there is no good way to select a sub-element of a view. The fact that an onClickListener is so awkward to implement because it is implemented as a typed class instead of a simple closure, made the situation so much worse. Finally, the fact that I had to set up all the events inside an activity’s onCreate (I mean, it can call a separate method that sets up the functions, but the idea is the same) was so awkward, it made me doubt whether I was doing something properly.

Enter Kotlin

The first solution I found was called Kotlin, which wasn’t a direct solution to the boilerplate problem, but made the syntax a lot more terse (though not necessarily better):

Switching to Kotlin resolves some of my peeves with Java.

  1. You don’t new a class because it is apparent that the keyword is useless when a class cannot be called like a function.
  2. No more button.setOnClickListener(new Button.OnClickListener(), Button button = new Button(), or type information that fundamentally just adds noise.
  3. The onCLickListener accepts what is visually a simple closure. It is just Kotlin’s magic, but you know. If it works, it works.

Enter Kotlin Synthetics

Since Jetbrains created both the Kotlin language and Android Studio, they have a lot of power in changing the way Android apps are created. To help solve the problem with “who the hell wants to use findViewById“, they created Kotlin Synthetics, a way to automatically inject views into the activity by their IDs. For example, if your button was defined in the XML as android:id="@+id/my_button", you will get a variable called my_button or myButton (I forgot which). Looking back, this really was a small improvement that added up as the number of buttons and textfields increased.

It was not a big transition to Jetpack view binding for the most part, other than a minor issue where nested views do not get bound the same way.

Enter Jetpack Compose

The year was (any time after 2013, really) and component-based architectures like React was all the rage. Soon after that, Facebook released ComponentKit, a UI framework for Objective-C heavily influenced by React. Apple released something called SwiftUI, which is more or less the same thing for Swift, their own language. What would Google do (WWGD)? Their answer is apparently to introduce Jetpack Compose, a way to build UI on top of their own programming language, for their own operating system, just like what Apple did.

Although the code appears to be longer than any of the examples above, it removes the need for a separate layout XML file, and event and data binding happen inline. These are major advantages of declarative UIs over imperative UIs, (with obvious drawbacks as well, but the benefits far outweigh the costs).

Enter me, using Jetpack Compose

Jetpack Compose is a mad attempt from Google at reimplementing React for Android, which suffers from the same problems as react: like how you are sometimes forced to do non-standard things in React, such as using className instead of class because the word class is taken, you will learn the quirks of Jetpack compose as you start to use it.

Modifier order matters?

In CSS, width: 100px; padding: 10px; it would not matter which order you place width and padding, as long as the two attributes do not interact with each other. In Jetpack Compose, they do matter. I can think of a thousand reasons why it would be better if the order does not matter, but let’s suppose this is a consequence of modifiers being actual method calls.

Google explained this behaviour as a feature:

The explicit order helps you to reason about how different modifiers will interact. Compare this to the view-based system where you had to learn the box model, that margins applied “outside” the element but padding “inside” it, and a background element would be sized accordingly.

This design is unintuitive to me. The two examples they gave demonstrates this:

The first example has a clickable padding because clickable was applied first. The second example does not have a clickable padding because the padding call was made before the clickable call. Does that make sense to you? I thought it would be the other way around. Leave a comment in the comment section down below. Remember to smash that like button and hit subscribe if you find this article useful. Or maybe it is subscribe first and like later. Who knows. Maybe order matters.

Modifiers are everything

Modifiers could conceivably be understood as CSS style="..." attributes, but not quite so. Unlike style, whose only task is to attach styles to an element, a Modifier are able to attach events as well.

This leads us to the next problem:

Every composable handles modifiers differently

This bit reminds me of HTML 4.01, when sometimes it is width="12", and sometimes it is style="width: 12px;".

Depending on the @Composable, it can accept either

  • No modifiers at all;
  • A modifier through the modifier parameter, which would allow you to add styling only;
  • A modifier through the modifier parameter, which would allow you to add events only;
  • A modifier through the modifier parameter, which would allow you to add both styling and events;
  • Individual parameters like onClick = () -> Unit that would allow you to add an event to it; or
  • Combinations of the above, but not using the standard parameter names onClick or modifier.

The issue is rampant even in the standard composables. For example, Text offers a color parameter, while a Card might use contentColor to the same effect. Card has a backgroundColor too. Does Text have that? No, it does not. How do you style a Text‘s background then, you ask? Well, you wrap a Suface around it, which accepts—wait for it—color, for its background colour.

The entire API screams, read the docs, and go fuck yourself on your way out.

Enter exit

All in all, using Jetpack Compose was not a joy in itself, but very much so when compared to the alternatives presented above. The library is currently at version 1.0.0-beta04, and Google developers have claimed to have frozen the API, so what we see now is what we will get for Compose V1. Let’s hope Compose V1 is at least usable right now, and Compose V2 fixes these issues without breaking in too many places.

The gold standard in value

Ever since Dicky got us off the Gold standard (albeit a bit unwillingly), the system has been rigged such that governments around the world can arbitrarily adjust the interest rate, and, therefore, the inflation rate (to a large extent). The price of gold has been going up ever since. But if you don’t own any gold, what good is it knowing how much it is?

What we really need to know is: how much our fiat (read: useless) money is worth, relative to the gold standard. For that, we generate an inversion of the gold price history—which no one on the Internet seemed to be interested in offering—and assume we have a thousand bucks to spend on gold.

In 1971, at \$35/oz, we could buy 28 whopping ounces of gold with $1000.

In 1972, your \$1000 is only good for 23 ounces.

In 1980, your “\$1000” can buy 1.1 ounces of gold. That is a 96% drop in buying power.

The buying power of useless money had recovered for a while, but it had gone downhill since 2000, when [Billy] signed some dumb paper.

So how much gold can you buy with a thousand bucks in 2021?

0.43 ounces. Your money had depreciated by 98%. The government stole 98% of your buying power (for gold), leaving you with just 2%.

So, if you were around in 1971, your investments today really should have gone up 50x, at the very least. If they did, then congratulations: you have successfully prevented the government from implicitly taxing you.

Perhaps a 50x “growth” over (2021-1971) = 50 years is a wonderful coincidence. However, the math is not that you simply “gain 1x every year”; you actually “multiply” your fortunes by 108% instead. A 5-8% gain per year is not unrealistic in the stock market, but rarely elsewhere (maybe real estate is a big hit these days). 1% bonds are the government’s way of telling you “you are a moron”, while it is unheard of for the stock market to consistently gain 5-8% every year.

“But but but inflation rate is only 650% over 50 years, not 5000%.” I hear you. Gold price did increase 600% between 1985 and 2021. However, 1985 was not when the gold standard was abolished. The difference between comparing with the gold standard, and comparing with inflation, is that the latter assumes consumer products have investment value. If the CPI were any good, we could possibly yield similar values for both.

Corporate dynamics

Someone who works at a big company shared his experience working there.

Your boss is not going to fire you because:

  • Your boss doesn’t pay you. That money comes from the heavens. It is not in his interest to save money.
  • Firing someone often negatively impacts team morale.
  • Once your position is gone, it could be gone forever. As a team leader from the higher-ups’ point of view, the bigger the team is, the more successful your boss is, as long as you (not them) get the job done.
  • If a series of layoffs is announced, you can act as buffer for the rest of the team, and be laid off then.
  • A company often issues shares to a team proportional to its size. If you are around doing nothing, a team of ten can be granted 1000 shares, but each person may get 111 shares while you get 0.
  • Ultimately, it is not your boss’s priority to make their team efficient. All it needs to do is to deliver a lot, to make themself look good so they get a promotion for managing a great team. Your boss will do nothing to you unless you get in their personal way.

You should be as lazy as your team’s average member.

  • Go out for lunch with your team. Take as long as you feel like. Take coffee breaks as well. Make sure you poop on company time. If you don’t do as they do, then you are telling the boss that everybody else is slacking off, and that is not a good look for the team or the boss.

Everyone will try to keep secrets from you because:

  • Everything you don’t know is something they do know. And if that something matters to the company (e.g. a critical process or know-how), then knowing that information makes you un-fire-able.

The lack of a method of (rightful) dismissal reduces the company’s ability to keep its employees on their toes. While there are always hardworking individuals who are self-motivated, being un-fire-able certainly encourages complacency of the lazy. Big companies systematically attract the less motivated, and that’s what we should be mindful of when finding a new place to work. Whether you personally find this desirable is down to personal preference.

Reasons to root in 2020

It is 2020. Most people have stopped bootloader unlocking and rooting their phones. Here are some of the things they say:

This is the first time I don’t root a phone since 2010 with android, and to be honest I don’t see the benefits of rooting anymore especially that (the skin) is great out the box.

I used ADB to be able to debloat entirely what I want. So don’t need to root for me.

All in all, there really aren’t a lot of reasons left. But there are till some:

Custom ROMs

If you cannot unlock your bootloader, your device will stop updating the moment the manufacturer decides you should get a new one. That should never be how it works. Imagine your toaster telling you to get a new one, even though it still works perfectly fine. Or imagine your computer telling you it can’t run some operating system. It doesn’t say that; it lets you run whatever you want as long as you have the compatible parts. Or imagine your home thermostat telling you it cannot work anymore.

Ad blocking

With root, you can edit your own /etc/hosts file or download an app to do it for you. No more ads in any app. If you are particularly adventurous and have an older device, you can even hide the ad spaces entirely so you don’t even know ads exist.

Without root, you can use some kind of VPN implementation I guess, but the VPN icon in the status bar will forever remind you that you are using some sort of workaround.

Firewall

With root, you can allow or deny any app internet access.

Without root, you rely—yet again—on an app establishing a VPN connection. It is still possible, but what if you don’t want to see ads either? Private DNS? That’s an awkward workaround that potentially exposes everything you do to a third party.

Call recording

What if you are allowed to record your own calls, but your operating system does not allow it? That’s just missing out on an important security feature.

Backup / Restore

Root can be used to access any app’s data directory. This allows you to backup the data and restore it at will.

Without root? If you must perform a hard reset, you data is done for. Some people may say “well I have backups to Google Drive enabled so I am fine.” Have you tried restoring from Google Drive? Half of it is broken.

System management

Did you know that you can turn charging on and off with root? You can write your own power management software to preserve your battery’s lifespan.

If you don’t have root, you are either stuck with whatever your manufacturer decided to implement, or literally nothing at all. And when your battery doesn’t last as long as it used to, you will have no choice but to fork out your own money for a replacement, knowing full well that, in fact, you could have done better.

Control / Replace Google Play Services

Complete system access allows you to remove Google Play Services entirely. If you don’t like or trust Google, this is the way to go. You can also replace Google Play Services with microG, so Google doesn’t phone (heh) home every ten seconds.

Without root? I am sorry. You should have known better.

Remove app permissions

Using root, you may revoke any app’s permissions (even if it breaks the app), disable the app, and even removing it permanently from storage. You can only do the second one without root.

Replace fonts / look and feel

On one hand, you can ask, “why on earth would you want to replace the system UI font?”

On the other hand, why not? What if you hate the font your phone comes with? What if you don’t want to put up with an eyesore everyday?

Turn the phone into a USB drive

Apps like DriveDroid, which turns your phone into a bootable USB drive, only work with root. Without that, your device will never appear bootable to a computer.

Timing the market

It is easy to convince someone out of timing the market. Take these four examples.

The trend is going down

Who knows when it will hit the true bottom. Don’t catch a falling knife, you know! Let’s wait and see.

The down trend reverses and it starts going up

It could be a dead cat bounce. Let’s wait until it’s actually going up.

The trend is actually going up now

It’s going up too quickly. Besides, the P/E ratio must be so high that we should not buy this stock right now. Remember: buy low, sell high. Better not buy now to avoid losing money.

The up trend reverses and starts going down

I told you it’s going to come back down. Let’s wait until the price is done dropping.

And then the stock continues the downtrend. All the while, any stocks you could have bought during the cycle might have gone up by 7% annually. If you want to lose out, just keep waiting…
