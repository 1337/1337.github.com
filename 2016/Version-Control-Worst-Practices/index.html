Version Control Worst Practices

Part of the PyCon series, by Greg Ward.

Commit any old crap

  • Sources of your libraries…
  • Built artifacts…
  • Build outputs…
  • Database from production…

Using version control as a backup tool

  • Except you backup (“commit”) whenever you want, not on a regular interval
  • Except you can ignore files for some reason
  • Except you need a message to justify why you’re backing up…

File sharing

  • Need to share a file? Commit it and push it on Github, so someone can pull it down!

Lone Wolf

  • Some developers can be such powerful version control users that their setup can discourage their setups from being used by any others.

Terrible commit messages

  • WIP
  • ongoing dev
  • .
  • working, pretty close (please come back)

Commit broken code

  • “Because it’s a backup tool! Better commit now!”

Commit nothing

  • “I can’t commit broken code!”

Committing unrelated changes

  • Switching branches / splitting changes is just too hard
  • Related: squashing changes

This makes commit history hard to understand.

Monolothic repository

  • Forcing your entire team (or worse, company) to share the same release schedule, version numbering, branching policy…
  • Even more fun when two teams want to merge into HEAD at the same time
  • Rebase hell

Too many repositories

  • One repository for each microservice is just too many
  • Should always put boilerplate code in its own repository, not have a copy in each repo

Presume infinite scalability

  • Remember version control systems also take time to do things
  • More commits means more time
  • Bigger files means more space
