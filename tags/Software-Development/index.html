The Saga Pattern

The Saga Pattern is a failure management pattern.

A saga is a long-lived transaction that can be written as sub-transactions. Each sub-transaction cannot take data from one another.

Each sub-transaction T has an idempotent “compensating transaction” C, that semantically undoes T. If T were sending an email, then C is a follow up email saying sorry.

The Saga Guarantee is

  • Either T1, T2, …, Tn all complete, or
  • T1, T2, …, Tn, Cn, …, C2, C1 all complete.

Implementing Saga in a distributed system relies on a component called the Saga execution coordinator (SEC). The SEC makes sure Cn is executed:

  • if Tn fails
  • if the Saga is aborted by some other T, or
  • if SEC fails. In this case, the restarted SEC runs C for any T in the Saga Log with a missing “end transaction” entry, then aborts the saga.

The Saga Log records what sub-transactions the SEC did in a Saga.
All sub-transactions are logged with Start and ends. See above for why it is done.

In distributed systems, “transactions” don’t exist, so T1, T2, …, Tn become Requests instead.

If Forward recovery (aka retrying) is desirable, then you need to make sure the act of retrying (i.e. T) is idempotent. At this point, you have essentially made an operation that can be started and stopped at any point, which can be delivered to multiple distributed services and succeed/fail independently.

Pragmatic Microservices for the Rest of Us

Part of the PyCon series, by Ryan Easterbrook.

“Well-made microservices can be rebuilt in a day”

The difference between services and microservices is that microservices only take care of one thing. If a service takes care of more than one thing, then it becomes harder to manage.

Microservices also tend to share the same standardised interface, which also help with managing them.

Example interface: REST

A simple microservice

A simple microservice consists of:

  • A shared “service base”, where all the boilerplate code is stored,
  • Service implementation, and
  • Protocol handler.

Database

None of the microservices will share the same database table.

If microservice A wants something from table B, it will ask microservice B for the information.

None of the microservices will share the same database table. This makes migrations massively easier to create, rollback, and squash. Having none of the microservices share the same table also means these tables can be taken out of the database with no difficulty, and can be scaled up using another instance very quickly.

Foreign keys

There will be no foreign keys. The code must manage the integrity.

Error handling

All microservice interfaces must be hard and well-defined.
If microservice A calls microservice B calls microservice C, and C raises an error (e.g. HTTP 500), then B must first catch all exceptions, and decide for itself whether to retry calling microservice C, or fail itself to microservice A as well.

Under no circumstance should microservice A receive any uncaught exceptions from microservice C.

Logging

Request logging is going to be easier per-microservice, but these logs themselves will make no sense. So, service request logs should be a thing.

Service Request Logs are trees of logs gathered to serve a single (web) request.

Having these logs also helps with performance tracking, to troubleshoot which microservices fail more often than expected.

Justification for microservices

Use microservice architecture if

  • you have multiple clients (or demos)
  • you have many developers, so they don’t need to rebase over each other’s stuff, or fight to merge into HEAD
  • you need to keep codebase complexity down, once and for all..

Arguments against microservices

  • Microservice code is cleaner only if your developers have the ability to write proper code. “Normal services” already encourage cleaner code, if developers have discipline.
  • Distributed services are never easy. Distributed transactions are even harder. No one will recover these errors for you, so you will need to manage each and every case.
  • Microservices are always slower than pure function calls, or a lower-numbered, normal services. If you want speed, try another language.
  • Testing microservices is easy. Testing integration among microservices just became massively more difficult.

Version Control Worst Practices

Part of the PyCon series, by Greg Ward.

Commit any old crap

  • Sources of your libraries…
  • Built artifacts…
  • Build outputs…
  • Database from production…

Using version control as a backup tool

  • Except you backup (“commit”) whenever you want, not on a regular interval
  • Except you can ignore files for some reason
  • Except you need a message to justify why you’re backing up…

File sharing

  • Need to share a file? Commit it and push it on Github, so someone can pull it down!

Lone Wolf

  • Some developers can be such powerful version control users that their setup can discourage their setups from being used by any others.

Terrible commit messages

  • WIP
  • ongoing dev
  • .
  • working, pretty close (please come back)

Commit broken code

  • “Because it’s a backup tool! Better commit now!”

Commit nothing

  • “I can’t commit broken code!”

Committing unrelated changes

  • Switching branches / splitting changes is just too hard
  • Related: squashing changes

This makes commit history hard to understand.

Monolothic repository

  • Forcing your entire team (or worse, company) to share the same release schedule, version numbering, branching policy…
  • Even more fun when two teams want to merge into HEAD at the same time
  • Rebase hell

Too many repositories

  • One repository for each microservice is just too many
  • Should always put boilerplate code in its own repository, not have a copy in each repo

Presume infinite scalability

  • Remember version control systems also take time to do things
  • More commits means more time
  • Bigger files means more space

The Bikeshed Effect

Ever had your friend proofread your thesis, and all you got for comments are things like “missing Oxford comma here” or “I think this word should be underlined”? As it turns out, there is a term for that: bikeshedding!

Bikeshedding describes the amount of time people spend on trivial things versus the non-trivial. Naturally, one ought to spend more time on the non-trivial because they are more difficult to understand; however, because so many people know about the trivial, they argue, over and over, about them. About the unimportant.

Wikipedia has its own meta article about its articles warning its editors against arguing about the trivial.

Bikeshed.com, a domain dedicated to this phenomenon, contains a sweet quote:

“The really, really short answer is that you should not. The somewhat longer answer is that just because you are capable of building a bikeshed does not mean you should stop others from building one just because you do not like the color they plan to paint it. This is a metaphor indicating that you need not argue about every little feature just because you know enough to do so. Some people have commented that the amount of noise generated by a change is inversely proportional to the complexity of the change.” - bikeshed.com

Some people argue that the trivial is not what the argument is about; many people believe that the argument is actually about game theory between the relative knowledge of the writer and the reader, while others think the ratio between trivial and non-trivial should strike a perfect balance to get the most out of a review.

To this day, I am still unsure how to tackle this problem at my workplace.
