Resetting Django Migrations

This is specifically meant for large web projects. For smaller, ordinary Django projects, see manage.py squashmigrations. For SQLAlchemy projects, do your own research.

Mechanism

Django 1.7+ stores migration information in a django_migrations table. The contents look something like

web=# select * from django_migrations;
id   |       app       |                      name                      |            applied            
-----+-----------------+------------------------------------------------+-------------------------------
12   | account         | 0002_email_max_length                          | 2017-03-21 14:16:17.046209-04
14   | admin           | 0002_logentry_remove_auto_add                  | 2017-03-21 14:16:17.098368-04
4    | auth            | 0002_alter_permission_name_max_length          | 2017-03-21 14:16:16.82983-04
788  | auth            | 0002_auto_...                                  | 2017-03-21 15:22:45.40872-04
573  | auth            | 0002_...                                       | 2017-03-21 15:06:12.735458-04
31   | authtoken       | 0002_auto_...                                  | 2017-03-21 14:16:19.556177-04

Running ./manage.py migrate inserts new entries into the table. Conversely, running ./manage.py migrate app {latest_version - n} removes entries from the table.

./manage.py squashmigrations

This command takes in a range of migrations from a specific app, combines the operations inside these migrations, and outputs one large migration with a replaces = [migration1, migration2, …] inside. This allows the list of migrations to be replaced by a larger one. Unless operations are cancelled out in migrations 1 to n, squashing migrations does not necessarily speed up the process.

From a series of experiments performed on a large web project, since squashmigrations operates on one app at a time, it does not appear to be effective on projects with many cross-app foreign keys, and unsquashable data migrations in between.

Manually clearing migration history

There are many questions online about resetting migrations, showing perhaps a need for large projects to reset migrations eventually.

Their procedures are usually as follows:

  1. Delete all existing migrations
  2. Fake migrations backward
  3. Create new migrations
  4. Fake migrations forward

Make sure current database state is consistent

./manage.py showmigrations

The above command should display a list with all migrations applied.

Fake migrations backward*

./manage.py migrate --fake APP_NAME1 zero
./manage.py migrate --fake APP_NAME3 zero
...

This fools Django into believing that the database is fresh. Double check all project apps have no migrations applied, and all third-party apps have all migrations applied, by running ./manage.py showmigrations again.

*This step might be completely necessary, but it should be done to be safe. It is also possible to replace these commands with a more intrusive DELETE FROM django_migrations WHERE app IN ('APP_NAME1', 'APP_NAME2', ...);

Delete all existing migrations

On a developer machine, delete all migrations in your “migrations” folders that are not __init__.py.

find . -path "*/migrations/*.py" -not -name "__init__.py" -delete

Create new migrations

Recreate all “effectively missing” migrations by running ./manage.py makemigrations.

If this step fails, the schema is too complex to be created in the first place. Consult an elder.

Ensure the new migrations are sane.

Create data migrations

Manually (and intelligently), port the data migrations in old migrations to new empty migrations on top of the new auto migrations. Some can be simplified; some are no longer necessary.

Test your changes

You now have a new set of auto migrations generated from the schema, and a ported list of data migrations that will be run after the schema is created.

Recreate your database, running all new migrations. The website should run as expected. Tests should pass as expected.

Applying changes on production / other machines

All existing instances of the project must have the latest migrations from develop applied.

Their procedures are usually as follows:

  1. Stay on old develop / master
  2. Fake migrations backward
  3. Switch to new develop / master
  4. Fake migrations forward

Faking migrations backward while on the old develop / master removes all records of them being applied.

Faking migrations forward while on the new develop / master pretends the new migrations are equivalent to the old ones.

After the operation, the Django project will not be aware of the old migrations. The transition is complete.

Python Core Devs Need You

Part of the PyCon series, paraphrasing whatever Raymond Hettinger said that day.

The core developers are not normal human beings

And they need to hear from you.

They just make the language and the standard library; they don’t know your use cases.

One time, a kind of averaging function made it into the standard library. It calculated the average to a billion digits of precision.

Well, as it turns out, that sort of precision isn’t meaningful to anyone using that function, so time is simply wasted calculating it.
If it is slow, then people won’t use it.
If no one will use it, then why is it in the standard library?

Python should be more friendly, and he could not count the ways

Throw a timestamp into a datetime function, and it says Expected a frggn buffer object. What does that even mean? You (the ordinary person) can definitely write more meaningful exception messages for the standard library.

How much better would it be, and how much time would it save everyone, if every error had a format like

That helps everyone, especially beginners.

The language is 25+ years old

list() works on a file but not on StringIO. Why is that? It is not intentional; the developers just didn’t have the time to add an equivalent to StringIO. If you want to help, this is one thing you can do.

Help update the docs

There are still parts of the standard library that are lacking docstrings. If you can accurately document these libraries, by all means!

Every time you submit even a trivial “fix”, it takes core devs an enormous amount of time

Just because you (the new developer) think something looks off, it doesn’t mean it should be “fixed”.
One example of such a story he told was someone changing all unit tests doing assert ... > ... to self.assertGreater(..., ...). Unbeknownst to the dude, some of these tests are actually supposed to test the bitwise shift operator (>), making some of the tests useless.
He also cautioned against refactoring other people’s tests. While code has tests to guard against mistakes, tests themselves do not, especially when some code was red-green developed, and refactoring the tests can potentially reduce the test coverage, or even give out false results.

It is pronounced the tupple

At least he’s saying it that way.

Advice for new developers

If you want to help, go do code reviews on the (Python) bug tracker or PEPs.

PEPs and bug trackers need critical thinking (i.e. knowing why something should not be implemented, rather than why something should).

Every Python feature request should result in a net reduction in complexity. It is easy to implement something. It is hard—extremely hard—to justify not implement it.

Work with developers who talk more about what they don’t know than what they do know.

If you learn without getting the impact, you’re working out wrong.

Understand what problems people are trying to solve, not what frameworks people are throwing at them all day long.

How to Make Mistakes in Python

tl;dr No wonder the book is free.

O’Reilly offered us this free eBook titled “How to make Mistakes in Python.” I am not sure if it was very good. Here are the key points.

Setup

  • Not using pip
  • Using the default REPL (Author recommended Jupyter but iPython is obviously superior)

Silly things

  • Not returning a value
  • Typos
  • Using def when you should have used class (which is 100% valid syntax)

Style

  • Using notationTheHungarian
  • Violating pep8
  • Stupid names like x and c or class TestImagineAClassNameThatExceeds80Characters
  • Complicated lambdas
  • Long ass-comprehensions

Structure

  • “Pathological if/elif blocks”, ie. Complicated flow control (cyclomatic complexity)
  • Useless getters and setters
  • Loads of decorators
  • Breaking the Law of Demeter, ie. Accessing attributes of arguments
  • Too many “__private” attributes
  • God objects and God methods
  • Things with global state

Surprises

  • Importing everything
  • Catching everything
  • Reinventing the standard library
  • Mutable argument defaults
  • Not using generators and wasting memory
  • Expecting tearDown is run in tests
  • Not logging
  • Not testing

These are really, really obvious mistakes that the apprentice developer would make, perhaps exclusively before they get a job, or before they finish high school. I hope they never happen to you.
